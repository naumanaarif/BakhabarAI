## 2. Non-negotiable principles

These rules can never be overridden by any spec, plan, or implementation:

### 2.1 Google Antigravity is the core orchestrator
- ALL agent workflows must be orchestrated through Google Antigravity
- No agent may run outside Antigravity's orchestration layer
- Every agent interaction must produce a traceable Antigravity log entry
- Use of external LLMs (Gemini, etc.) is allowed only as tools called
  from within Antigravity — not as standalone orchestrators

### 2.2 Multi-agent architecture is mandatory
- The system must implement exactly 5 named agents (see Section 5)
- Each agent must be distinctly named and identifiable in logs
- Agent logs must show: agent name, input received, reasoning steps,
  tool calls made, output produced, confidence score, timestamp
- No agent may be collapsed into another — judges will verify separation

### 2.3 Mobile app is the primary deliverable
- Flutter mobile app is mandatory for submission
- All citizen-facing features live in the Flutter app
- The app must be buildable to APK for submission
- Web app/dashboard is optional — do not prioritise over Flutter

### 2.4 Data privacy
- No real personal data may be stored or transmitted
- All citizen reports use anonymised/mock identifiers
- Phone numbers stored only as hashed tokens after OTP verification
- Mock datasets must be clearly labelled as mock in documentation

### 2.5 Simulation is not optional
- At least one action must be fully simulated end-to-end
- Simulation must show visible before-state → action → after-state
- The false alarm / alert retraction scenario must be demonstrable
- Baseline comparison (rule-based vs agentic) must be implemented

### 2.6 Robustness evidence is required for submission
- At least one of these must be demonstrable in the app:
  - False alarm detected and alert retracted
  - API failure with fallback to cached data
  - Conflicting signals resolved with reasoning
  - Missing location data handled gracefully

## 5. Agent definitions — immutable

These agent names, responsibilities, and output contracts must not change.
All specs and implementations must reference agents by these exact names.

### Agent 1 — SignalHarvester
**Responsibility:** Ingest and normalise multi-source signals  
**Inputs:**
- Citizen reports from Flutter app (incident type, location, description, media)
- Mock weather API response (rainfall mm/hr, heat index °C, humidity %)
- Mock Maps/traffic API response (congestion % by road segment)
- Mock 1122/PSCA emergency call feed (call count, location, type)
- Mock social media posts (text, geotag, timestamp) — include Roman Urdu

**Outputs:**
- Normalised signal bundle (JSON)
- Credibility score per signal (0.0–1.0)
- Noise flag for irrelevant signals (boolean)
- Source type label per signal

**Rules:**
- Must handle Roman Urdu text without crashing
- Must deduplicate signals from the same event within a 5-minute window
- Must label at least 1 signal as noise in every demo scenario
- Confidence score must appear in Antigravity log

---

### Agent 2 — CrisisDetector
**Responsibility:** Classify crisis type, severity, and confidence  
**Inputs:** Normalised signal bundle from SignalHarvester

**Outputs:**
- Crisis type (flood / heatwave / traffic / infrastructure / other)
- Severity level (critical / elevated / routine)
- Confidence score (0.0–1.0)
- Affected radius (km)
- Affected population estimate
- Expected duration (hours)
- Conflicting signal flag (boolean) — triggers false alarm check

**Rules:**
- Must support simultaneous detection of 2+ crisis types
- Must produce a false alarm output in the retraction demo scenario
- Confidence score below 0.5 must trigger a verification step, not immediate alert
- All outputs must be structured JSON — no free text only responses

---

### Agent 3 — SituationReasoner
**Responsibility:** Cross-link crises, predict cascades, produce compound risk score  
**Inputs:** Crisis objects from CrisisDetector + resource availability from Firestore

**Outputs:**
- Compound risk score (0–100)
- Cascade prediction (e.g. "flooding blocks routes to cooling centres")
- Impact analysis narrative (plain English, ≤3 sentences)
- Priority ranking of simultaneous crises
- Recommended action brief for ResourcePlanner

**Rules:**
- Must explicitly reason about the relationship between simultaneous crises
- The cascade prediction is the key judge differentiator — never skip it
- Plain English narrative must be displayable in the Flutter Expert View
- Must handle single crisis scenario (no cascade) gracefully

---

### Agent 4 — ResourcePlanner
**Responsibility:** Allocate constrained resources, generate action plan  
**Inputs:** Recommended action brief from SituationReasoner + resource pool from Firestore

**Resource pool (mock data):**
- Rescue boats: 12 total
- Ambulances: 45 total
- Police units: 120 total
- Rescue teams: 30 total
- Shelters: 8 locations with capacity
- Water tankers: 6 total

**Outputs:**
- Prioritised action list (3–5 actions per crisis)
- Resource allocation per action (units assigned, cost, ETA)
- Trade-off explanation when resources are shared across crises
- Stakeholder notification drafts (public, hospitals, NDMA)

**Rules:**
- Must show resource trade-offs when 2 crises compete for same pool
- Must attach constraints to each action (budget limit, time limit, availability)
- Stakeholder messages must be audience-specific — public gets plain Urdu/English,
  NDMA gets technical summary
- Must produce a baseline comparison output (rule-based vs agentic allocation)

---

## 5. Agent 5 — ResponseExecutor
**Responsibility:** Simulate execution of actions, push results to Flutter app  
**Inputs:** Action plan from ResourcePlanner

**Simulated executions:**
- Traffic rerouting: update mock route JSON in Firestore
- Rescue dispatch: write boat/ambulance assignments to Firestore
- Shelter activation: update shelter capacity records in Firestore
- SMS/push alert: send FCM notification to Flutter app
- NDMA escalation: create mock ticket in Firestore
- Alert retraction: update alert status to "retracted" + send correction FCM

**Outputs:**
- Execution log per action (action, timestamp, status, result)
- Before-state snapshot
- After-state snapshot
- Overall response progress % (actions complete / total)
- Risk score after intervention

**Rules:**
- Every simulated action must write a visible state change to Firestore
- Alert retraction must be demonstrable — required for robustness evidence
- Before/after risk scores must appear in Flutter Simulation screen
- Execution logs must match the Antigravity trace logs submitted for judging

| P2 | Resource allocation screen (read-only citizen / editable expert) | Agent 4 output |

### Expert View toggle rules
- Toggle lives only on the Incident Detail screen
- OFF state (citizen): shelter location, safe route, emergency number, AI Verified badge
- ON state (expert): agent trace, simulation actions, resource allocation, raw confidence scores
- Topbar tints dark red (#2A0A0C) when expert mode is active
- State is session-only — resets on app restart

### Citizen-facing language rules
- Raw confidence scores (e.g. 0.92) must never appear in citizen view
- Replace with: "AI Verified" (≥0.75), "AI Monitoring" (0.5–0.75), "Unverified" (<0.5)
- Incident types on home screen must be crisis events only:
  flood / heatwave / traffic gridlock / power outage / infrastructure failure
- "Unauthorized Crowd Gathering" and "Suspicious Vehicle Loitering" are
  NOT valid crisis types for this app — do not use them anywhere

## 7. Demo scenario — canonical

This is the scenario used in all mock data, demos, and the submission video.
All agents must handle this scenario correctly before submission.

**Scenario:** Karachi monsoon compound crisis

**Input signals:**
1. Twitter: "G-10 mein pani bhar gaya, gaariyan phans gayi hain" (credibility: 0.82)
2. Twitter: "Rashid Minhas Road completely blocked" (credibility: 0.78)
3. Weather API: Rainfall 82mm/2hr, heat index 48°C (credibility: 0.97)
4. Maps API: Congestion +340% on Rashid Minhas Rd (credibility: 0.95)
5. 1122: 12 rescue calls in 20 min, 3 stranded vehicles (credibility: 0.99)
6. Twitter: "PSL match crowd noise in Gulberg" (noise — irrelevant, filtered)
7. Twitter: "Flash flood in Saddar" (conflicting — later confirmed water main burst)

**Expected agent outputs:**
- SignalHarvester: 6 valid signals ingested, 1 filtered as noise
- CrisisDetector: Flood (G-10, confidence 0.92, critical) + Heatwave (Korangi, confidence 0.78, elevated)
- SituationReasoner: Compound risk 87/100. "Flooding in G-10 blocks 2 of 3 routes to Korangi cooling centres. Heat stress on displaced population elevates medical emergency risk."
- ResourcePlanner: 5 actions generated, resources allocated across both crises
- ResponseExecutor: 5 actions simulated, risk drops to 34/100, false alarm retracted for Saddar

**False alarm scenario (required):**
- Signal 7 (Saddar flood) initially triggers low-confidence alert (0.48)
- Field verification confirms water main burst only
- CrisisDetector updates classification
- ResponseExecutor retracts public alert, sends correction FCM
- Retracted alert shown with strikethrough in Flutter alerts screen

## 9. Cost and scalability notes (for README)

- Gemini 2.0 Flash used for all agent calls — lowest cost tier
- Estimated cost per full pipeline run: < $0.01
- Firestore reads/writes: < 100 per incident cycle
- At 10x scale (10 simultaneous incidents): agent parallelisation via
  Antigravity multi-agent coordination
- At 100x scale: Vertex AI autoscaling + Firestore horizontal scaling
- Latency target: signal → alert < 30 seconds for P0 crises
